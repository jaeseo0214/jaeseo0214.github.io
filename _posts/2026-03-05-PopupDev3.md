# [트러블슈팅]


## 1. JPA 기본 생성자 (Entity 부분)
<br>


[문제 상황]<br><br>


![트러블슈팅1](/assets/images/트러블슈팅1.jpg)
<br>


* 현상: 서버를 띄우고 위와 같은 오류 발생<br>


* 로그 확인: 로그 확인 결과 다음과 같은 에러 메시지를 확인<br>

```java
org.hibernate.InstantiationException: No default constructor for entity 'com.project.popupstore.entity.PopupStore'
```
<br>


---


[원인 분석]<br><br>


* 분석: 에러 메시지는 PopupStore 엔티티에 기본 생성자가 없음을 지적하고 있었음<br>


* 이론적 배경:<br>

    * JPA(Hibernate)는 데이터베이스에서 조회한 데이터를 자바 객체로 변환할 때,<br>
    리플렉션(Reflection)이라는 기술을 사용함<br>


    * 리플렉션은 클래스의 이름만 알고 있다면 생성자, 메소드 등에 접근할 수 있게 해줌<br>
    이때, 가장 먼저 호출하는 것이 파라미터가 없는 기본 생성자임<br>


    * 난 id, name, 위/경도 값을 받기 위해 매개 변수가 있는 생성자를 정의했음<br>
    이로 인해 자바 컴파일러가 자동으로 만들어주는 기본 생성자가 사라진 게 원인이었음<br>
<br>


---


[해결 방법]<br><br>


* 단순 해결: 처음에는 단순히 public으로 기본 생성자를 추가해 줬음<br>


* 설계 고민: 하지만 접근 제어자를 public으로 설정하면,<br>
프로젝트 어디서든 new PopupStore()와 같은 방법으로 값이 비어있는<br> 
불완전한 객체가 생성될 위험이 있다고 판단함<br>


* 최종 해결: 접근 제어자를 protected로 변경 -> JPA 프레임워크는 접근할 수 있으면서<br> 
외부의 잘못된 객체 생성은 제한함 -> 객체의 캡슐화를 유지<br>


```java
@Entity
public class PopupStore {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) 

    private Long id;
    private String name;
    private double latitude;
    private double longitude;
    private String category;

    // 기본 생성자 <------- 이 부분!
    protected PopupStore() {
    }

    public PopupStore(Long id, String name, double latitude, double longitude, String category) {
        this.id = id;
        this.name = name;
        this.latitude = latitude;
        this.longitude = longitude;
        this.category = category;
    }
}
```
<br>


---


[배운점]<br><br>


* 프레임워크를 사용할 땐 단순 사용하는 것에 그치는 것이 아니라,<br>
그 프레임워크가 내부적으로 어떻게 객체를 생성하고 관리하는지를<br>
이해하는 것이 중요하다는 것을 깨달음<br>


* 또한 에러를 해결하면서 `객체지향적인 설계`를 함께 고민해야겠다는 생각을 함<br>
<br>






## 2. 사용자의 다양한 입력에 대비한 방어적 설계 (Controller 부분)
<br>


이 부분은 실제 오류라기보다는,<br>
사용자가 다양한 방식으로 요청을 보낼 수 있는 상황을 고려하여<br>
서버가 안정적으로 동작하도록 하기 위한 방어적 설계이다.<br><br>


---


[고민 지점]<br><br>


* 의문1: 서버에서 요구하는 파라미터가 누락된 채 요청이 들어온다면?<br>

    * 출발 좌표는 필수지만, 선택한 팝업스토어 목록은<br> 
    사용자가 선택하지 않을 수도 있는 데이터임<br> 
    이 경우 서버가 어떻게 반응해야 할지 고민함<br><br>


* 의문2: 빈 ID 목록이 서버로 전달될 때, 굳이 DB를 조회해야 할까?<br>

    * 사용자가 아무것도 선택하지 않았을 때,<br> 
    에러는 나지 않더라도 굳이 DB 접속 비용을 써가며<br> 
    쿼리를 날리는 것은 자원 낭비라고 판단함<br>
<br>


---


[해결 방법]<br><br>


* 해결1: stores 파라미터에 `required = false` 옵션을 설정하여<br>
해당 값이 요청에 포함되지 않았을 때 발생할 오류에 대비함<br>


* 해결2: ID 목록이 비어 있는 경우에는<br>
DB 조회를 수행하지 않고 `Collections.emptyList()`를 사용하여<br>
바로 빈 리스트를 반환하도록 처리함<br>


```java
// 동선 관련 계산
    @GetMapping("/route")
    public RouteResult getOptimalRoute(
            @RequestParam double startLat,
            @RequestParam double startLng,
            @RequestParam(value = "stores", required = false) List<Long> storeIds 
    ) {
        List<PopupStore> targetStores;

        // 빈 ID 목록일 경우 빈 리스트 반환
        if (storeIds == null || storeIds.isEmpty()) {
            targetStores = Collections.emptyList();
        } else {
            // 모든 데이터가 아닌, 전달받은 ID에 해당하는 데이터만 서비스에서 가져옴
            targetStores = service.getStoresByIds(storeIds);
        }

        return service.calculateOptimalRoute(startLat, startLng, targetStores);
    }
```
<br>


---


[배운점]<br><br>


* 컨트롤러에서 단순 요청만 처리하는 것이 아닌,<br>
사용자의 다양한 입력 형태를 고려하여<br>
서버가 안정적으로 동작할 수 있도록<br>
방어적으로 코드를 설계하는 것이 중요하다는 것을 생각하게 됨<br>


* 기능 동작에만 집중하기보다,<br>
불필요한 DB 조회와 같은 자원 낭비를 줄이는 방식으로<br>
코드를 설계하는 것 역시 중요하다는 것을 체감함<br>

# [개선 계획]


## 실제 도보 기반 길 안내


![사용 화면](/assets/images/작동2.png)


현재 시스템은 위와 같이 두 지점 사이의 직선 거리를 기반으로 최적 순서를 계산하고 있다.<br>

하지만 실제 사용자가 길을 찾을 때는<br> 
건물의 위치나 도로의 형태에 따라 이동 거리가 크게 달라질 수밖에 없다.<br>

현재의 방식은 이런 실제 보행 환경을 반영하지 못해<br> 
서비스의 실용성이 다소 부족하다.<br>

이를 개선하기 위해 TMAP API(보행자 경로 안내) 등을 연동하여,<br> 
실제 골목길이나 횡단보도 정보를 반영한<br> 
`도보 기반 소요 시간 및 이동 거리`를 계산하고 보여줄 예정이다.<br> 
이를 통해 더욱 정확한 경로를 사용자에게 제공하는 서비스를 만들 예정이다.<br>
