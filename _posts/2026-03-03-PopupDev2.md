# [개발 기록]


이번 프로젝트는 단순히 서비스를 만드는 것을 넘어,<br> 
백엔드 개발의 기초를 제대로 다지는 데 목적이 있다.<br> 
개발 과정에서 마주친 고민과 해결 과정을 기록으로 남긴다.<br>





## [시스템 구조]


사용자가 요청을 보내서 결과를 받기까지 데이터가 어떻게 흐르는지 정리했다.<br>


![프로젝트 구조](/assets/images/프로젝트%20구조.jpg)


1. 사용자: "내 위치는 여기고, A, B, C 팝업스토어를 가고 싶어"라고 요청을 보냄.<br>

2. 컨트롤러(Controller): 요청을 가장 먼저 받아서 확인하는 역할.<br>

3. 서비스(Service): 프로그램의 두뇌. 여기서 '계산기'를 돌리기 위해 데이터를 가져오라고 명령함.<br>

4. 레포지토리(Repository): DB에 가서 사용자가 선택한 장소들의 진짜 좌표(위도, 경도)를 찾아 서비스에 전달함.<br>

5. 응답(Response): 서비스가 계산한 '방문 순서'와 '총 거리'를 하나로 묶어(JSON) 사용자에게 돌려줌.<br>



## [개발 상세 과정]


### 1. [데이터 설계 - 정보 저장 및 관리]


가장 먼저 팝업스토어 데이터를 담을 그릇인 Entity와<br> 
데이터에 접근하기 위한 Repository를 설계했다.<br>


[Entity 설계]<br>

```java
@Entity
public class PopupStore {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // DB가 자동으로 ID 부여
    private Long id;
    private String name;
    private double latitude;  // 소수점이 긴 위도/경도를 위해 double 선택
    private double longitude;
    private String category;

    protected PopupStore() {} // JPA에서 필수인 기본 생성자
}
```
<br>


[Repository 설계 - Spring Data JPA 활용]<br>

```java
public interface PopupStoreRepository extends JpaRepository<PopupStore, Long> {
}
```

### 2. [알고리즘 구현 - 최적의 순서를 찾는 로직]


이 프로젝트의 가장 핵심적인 부분이라고 할 수 있다.<br><br>


[방문 순서 결정: 탐욕 알고리즘(Greedy)]<br>

모든 경로를 다 따지면 계산이 너무 복잡해지기 때문에,<br> 
"현재 위치에서 가장 가까운 곳을 가고,<br> 
거기서 다시 그다음 가까운 곳을 찾는" 방식을 선택했다.<br>

* why? -> 성수동 팝업스토어는 개수가 많아야 수십 개이므로,<br> 
데이터가 적은 상황에서는 연산이 빠르고 코드가 직관적인<br> 
Greedy 방식이 가장 효율적이라고 판단했다.<br><br>


[거리 계산: 하버사인(Haversine) 공식]<br>

지구는 평면이 아니라 곡면이기 때문에 단순 평면 계산 공식이 아닌,<br> 
구면 좌표계를 반영한 공식을 사용했다.<br>


```java
// 하버사인 공식을 이용한 단순 거리 계산 메소드
   private double calculateDistance(double lat1, double lng1, double lat2, double lng2) {

        double latDistance = Math.toRadians(lat2 - lat1);
        double lonDistance = Math.toRadians(lng2 - lng1);

        double a = Math.sin(latDistance / 2) * Math.sin(latDistance / 2)
                + Math.cos(Math.toRadians(lat1)) * Math.cos(Math.toRadians(lat2))
                * Math.sin(lonDistance / 2) * Math.sin(lonDistance / 2);

        double c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));

        return EARTH_RADIUS * c;
    }


 // 가까운 곳을 순차적으로 찾는 방문 순서 메소드
    private List<PopupStore> calculateVisit(double startLat, double startLng, List<PopupStore> stores) {

        // 결과 리스트
        List<PopupStore> result = new ArrayList<>();

        // 남은 방문 대상 리스트
        List<PopupStore> nonVisitList = new ArrayList<>(stores);


        // 시작 위치 설정
        double currentLat = startLat;
        double currentLng = startLng;

        while (!nonVisitList.isEmpty()) {

            PopupStore nearest = null;
            double minDistance = Double.MAX_VALUE;

            for (PopupStore store : nonVisitList) {

                double distance = calculateDistance(currentLat, currentLng, store.getLatitude(), store.getLongitude());

                if (distance < minDistance) {
                    minDistance = distance;
                    nearest = store;
                }
            }

            result.add(nearest);
            currentLat = nearest.getLatitude();
            currentLng = nearest.getLongitude();

            nonVisitList.remove(nearest);
        }

        return result;
    }
```
<br>


[결과 포장 (RouteResult)]<br> 

단순히 순서 리스트만 주는 게 아니라,<br>
"총 이동 거리"까지 한 번에 알려주기 위해 정보를 묶어서 반환하는 구조를 설계했다.<br>


```java
// dto
public class RouteResult {

    private List<PopupStore> visitOrder;
    private double totalDistance;

    public RouteResult(List<PopupStore> visitOrder, double totalDistance) {
        this.visitOrder = visitOrder;
        this.totalDistance = totalDistance;
    }
}
```
<br>


* [무엇을 배웠나?]<br>

    * 단순히 기능을 만드는 게 아니라, 응답을 받는 프론트엔드 입장에서 데이터를 어떻게 묶어줘야 편할지 고민하며 DTO 설계의 중요성을 배움.<br>


    * 리스트(ArrayList)를 생성하고 관리하며 조건에 맞는 데이터를 필터링하는 기초 자료구조 활용 능력을 다짐.<br>

### 3. [컨트롤러 연결]


구현된 로직을 외부에서 사용할 수 있도록 컨트롤러를 구축했다.<br>


```java
@RestController

@RequestMapping("/popupstores")
public class PopupStoreController {


    private final PopupStoreService service;

    public PopupStoreController(PopupStoreService service) {
        this.service = service;
    }


    // 팝업스토어 저장
    // RequestBody 는 JSON -> 객체로 변환
    @PostMapping
    public PopupStore addPopupStore(@RequestBody PopupStore popupStore) {
        return service.addPopupStore(popupStore);
    }


    // 모든 팝업스토어 리스트 조회
    @GetMapping
    public List<PopupStore> getAllPopupStores() {
        return service.getAllPopupStores();
    }

    // 동선 관련 계산
    @GetMapping("/route")
    public RouteResult getOptimalRoute(
            @RequestParam double startLat,
            @RequestParam double startLng,
            @RequestParam(value = "stores", required = false) List<Long> storeIds // ID 리스트 추가
    ) {
        List<PopupStore> targetStores;

        if (storeIds == null || storeIds.isEmpty()) {
            targetStores = Collections.emptyList();
        } else {
            // 모든 데이터가 아닌, 전달받은 ID에 해당하는 데이터만 서비스에서 가져옴
            targetStores = service.getStoresByIds(storeIds);
        }

        return service.calculateOptimalRoute(startLat, startLng, targetStores);
    }
}
```
<br>


[데이터 선택 조회]<br>

자원 낭비를 줄여보고자 DB 전체 조회가 아닌,<br> 
선택된 ID 리스트만 필터링하여 가져오도록 설계했다.<br><br>


* [무엇을 배웠나?]<br>

    * @RestController와 @RequestMapping을 통해 특정 요청을 처리하는 엔드포인트 설계 방식을 익힘.<br>


    * @RequestParam의 required = false 옵션과 null 체크를 통해 사용자의 다양한 입력 상황에 대응하는 방어적 코딩의 중요성을 깨달음.<br>


    * 컨트롤러는 요청 수신과 응답에만 집중하고, 복잡한 비즈니스 로직은 서비스에 위임하는 계층 분리의 이점을 체감함.<br>
